Drift + Dendric

A complete agentic primitive suite — intent-based language, biologically-inspired memory, and adaptive voice architecture. Designed together from first principles.

01 — Vision

The Thesis

We are in the PHP/CGI era of agentic programming. Every team building AI agents is hand-rolling the same infrastructure: prompt construction, model selection, output parsing, retry logic, cost tracking, memory management. The abstractions haven't crystallised yet. Drift + Dendric is a bet on what those abstractions should be.

The core insight: agent programs are fundamentally different from traditional programs. They are non-deterministic, they have cost, they fail gracefully, they remember, they learn, they compose. A truly agent-native system makes all of these structural rather than opt-in.

Drift is how an agent thinks — an intent-based programming language that transpiles to Python.
Dendric is how an agent remembers — a biologically-inspired memory engine with four retrieval paths, temporal decay, and spreading activation.
Together they form a cognitive architecture for agents, expressed as a programming language.

Design Principles

Intent over instruction

Say what, not how

classify doc as FitScore transpiles to ~40 lines of Python with prompting, validation, retries, and cost tracking. The programmer declares intent. The toolchain handles execution.

Cost as first-class citizen

Budget in the grammar

Every agent has a cost cap enforced at the language level. Not an afterthought. Not a library you remember to import. Structural, like type safety.

Transparent compilation

Readable output

Generated Python is clean, documented, and debuggable. No black box. The same .drift input always produces the same .py output. The transpiler is deterministic — no AI in translation.

Progressive disclosure

Readable by everyone

Non-engineers read .drift files and understand the agent logic. Engineers drop into the generated Python for full control. Domain experts can modify agent behaviour without touching code.

02 — Drift Language

Language Overview

Drift is an intent-based programming language for agentic AI systems. Programs describe what an agent should accomplish rather than how to accomplish it. The Drift transpiler converts .drift files into production-grade Python 3.11+ code that calls the Drift runtime.

Core Primitives

Intent expressions are the core Drift primitive — they replace 30-50 lines of Python with a single declarative statement:

Model Routing

The model router selects the optimal model per-call based on cost, availability, and task complexity. Budget-aware degradation means the agent automatically shifts to cheaper models as the budget cap approaches.

}
Complete Example — Grant Eligibility Checker 
schema FitScore {
  company_name: string
  overall_score: number between 0 and 100
  matching_criteria: list<string>
  gaps: list<string>
  recommendation: one of "strong fit", "possible fit", "weak fit", "no fit"
  confidence: number between 0 and 1
}

agent GrantChecker {
  model: prefer "claude-sonnet" fallback "gpt-4o"
  budget: £5 per run
  memory: dendric("grant_history")

  step evaluate(company: document, call: document) -> FitScore {
    recall past_scores from memory using company

    let score = rate company against call
      using past_scores
      as FitScore

    deja_vu match on score {
      "similar_rejection" -> {
        respond "Warning: similar profile was rejected for {match.grant_title}"
      }
    }

    remember score tagged company.sector, call.funder
    return score
  }
}

03 — Dendric Memory Engine

Biologically-Inspired Memory for Agents

Dendric is a memory lifecycle engine modeled on biological memory systems. It provides four parallel retrieval paths fused via Reciprocal Rank Fusion (RRF), neurotransmitter-modeled encoding dynamics, sleep-cycle consolidation, and temporal decay. Benchmarked on LongMemEval showing +6.7pp recall over vector-only baseline on aged-corpus retrieval.

Four Retrieval Paths

Vector

Semantic Similarity

Embedding-based retrieval. Finds memories that are semantically related to the query even without keyword overlap. The standard path for fuzzy matching.

FTS5

Full-Text Search

SQLite FTS5 keyword matching. Fast exact-match retrieval. Catches memories that share specific terms, names, or identifiers that embeddings might miss.

Entity Graph

Structural Traversal

Knowledge graph of entities and relationships. Traversal-based retrieval finds memories connected through entity chains: Company → Sector → Funder → Past Application.

Associative

Spreading Activation

Biologically-inspired activation propagation. Nodes activate their neighbours along weighted edges. Finds memories through indirect, non-obvious connections.

RRF Fusion

All four paths run in parallel. Results are fused using Reciprocal Rank Fusion — each result's final score is the sum of 1 / (k + rank) across all paths that returned it. This means a memory ranked highly by multiple paths scores much higher than one ranked #1 by a single path. The parameter k (default 60) controls how much weight is given to high-rank positions versus appearing in multiple paths.

Neurotransmitter Model

Four scalar parameters modulate memory encoding and retrieval dynamics:

ParameterBiological AnalogEffect on Agent
DopamineReward signalStrengthens edges associated with successful outcomes. High after positive feedback or correct predictions.
NorepinephrineAlertnessLowers recognition threshold for deja_vu. High during critical tasks. Makes the agent more likely to notice connections.
AcetylcholineAttention/focusFocuses activation propagation along task-relevant edges rather than spreading diffusely. Narrows retrieval scope.
SerotoninStabilityModulates graph restructuring aggressiveness. High serotonin = conservative consolidation, preserving existing structure.

Implementation note: these are not literal neurotransmitter simulations. They are four parameterised scalars that modulate edge weights, activation thresholds, propagation breadth, and consolidation intensity. The biological framing is the conceptual model; the implementation is parameterised graph dynamics.

Temporal Decay & Consolidation

Memories decay over time using a hybrid exponential + power-law curve. Frequently-accessed or strongly-encoded memories decay slower (the "use it or lose it" principle). Consolidation cycles run at agent boundaries — when a pipeline completes, short-term step outputs merge into long-term memory, weak memories are pruned, and entity graph connections are strengthened or weakened based on co-occurrence.

Five Keywords for Agent Memory

The Drift language exposes Dendric through five memory primitives. Together they cover the complete lifecycle of agent memory: encoding, retrieval, recognition, organisation, and forgetting.

remember — Intentional encoding

The agent has learned something and wants to persist it. An explicit, intentional store — the agent knows this matters. Triggers encoding with entity graph linking and high dopamine encoding strength.

-- Basic remember
remember score tagged "healthcare", "innovate_uk"

-- Remember with entity links from the data
remember score tagged company.sector, call.funder

-- Hierarchical context
remember analysis tagged project.domain > project.sub_domain > task_type

Transpiles to:

# Dendric encoding with entity linking
await self.memory.encode(
    data=score,
    tags=["healthcare", "innovate_uk"],
    encoding_strength=self._agent_state.dopamine,
    entity_links=self.memory.graph.extract_entities(score),
)
recall — Active retrieval

The agent deliberately searches its memory for relevant context. Conscious retrieval — the agent knows it needs information and goes looking. Triggers RRF-fused four-path search.

-- Basic recall
recall past_evaluations from memory using company_profile

-- Scoped recall with context tags
recall past_scores from memory
  tagged "innovate_uk"
  using company_profile

-- Recall with limit
recall recent_cases from memory using query limit 5

Transpiles to:

# Dendric RRF-fused multi-path retrieval
past_evaluations = await self.memory.recall(
    query=company_profile,
    tags=None,
    limit=10,
    paths=["vector", "fts5", "entity_graph", "associative"],
    rrf_k=60,
)
deja_vu — Involuntary recognition

The defining primitive. The agent didn't ask to remember. It just noticed. The spreading activation network fires in the background, continuously comparing incoming patterns against stored patterns. When something crosses the recognition threshold, the agent becomes aware.

Key distinction: recall is "go find me X." deja_vu is "if anything in memory resonates with what I'm currently looking at, surface it." Reactive, not proactive. 
agent GrantChecker {
  memory: dendric("grant_history")

  step evaluate(company: document, call: document) -> FitScore {
    let score = rate company against call as FitScore

    -- The agent recognizes a pattern it's seen before
    deja_vu match on score {
      "similar_rejection" -> {
        respond "Warning: similar profile was rejected for {match.grant_title}"
        let score = score with { confidence: score.confidence * 0.7 }
      }
      "successful_pattern" -> {
        respond "Pattern matches previous successful application"
      }
    }

    return score
  }
}

How deja_vu works at the engine level:

  1. Entity extraction — extract seed entities from the current step context
  2. Seed activation — activate corresponding nodes in the entity graph
  3. Bounded propagation — spread activation 1-2 hops along weighted edges (milliseconds, not seconds)
  4. Threshold modulation — norepinephrine level adjusts recognition sensitivity
  5. Pattern matching — if any non-seed node crosses the threshold, retrieve its associated memory
  6. Classification — classify the match type ("similar_rejection", "successful_pattern") for the match block

Transpiles to:

# Passive recognition check via Dendric spreading activation
_deja_vu_result = await self.memory.deja_vu.check(
    context=score,
    agent_state=self._agent_state,
)

if _deja_vu_result is not None:
    match = _deja_vu_result
    if match.pattern_type == "similar_rejection":
        self.output(f"Warning: similar profile was rejected for {match.memory.grant_title}")
        score = replace(score, confidence=score.confidence * 0.7)
    elif match.pattern_type == "successful_pattern":
        self.output("Pattern matches previous successful application")
tagged — Context organisation

Tags organise memories into the entity graph. They become nodes with edges to the memories they label. Retrieval can scope by tags, and spreading activation traverses tag connections — a recall on "healthcare" can pull in NHS-related memories through graph traversal even if they weren't explicitly tagged with "healthcare".

-- Flat tags
remember result tagged "healthcare", "innovate_uk"

-- Dynamic tags from data
remember result tagged company.sector, call.funder

-- Hierarchical (creates graph path)
remember result tagged domain > sub_domain > task_type

-- Filter recall by tags
recall cases from memory tagged "innovate_uk" using query
forget — Deliberate pruning

Agents accumulate context that becomes stale or irrelevant. forget triggers accelerated decay in Dendric — instead of waiting for natural consolidation to prune weak memories, the agent deliberately marks memories for removal.

-- Explicit target
forget memories tagged "deprecated_criteria"

-- Age-based pruning
forget memories older than 90d tagged "market_data"

-- Pattern-based
forget memories where confidence < 0.3
05 — Adaptive Knowledge Graph

Dynamic Graph Architecture for deja_vu

Standard knowledge graphs are passive data structures — you query them, you get results. deja_vu requires an active graph that does work even when nobody asked it a question. Three capabilities beyond standard graph databases:

1. Continuous Activation Propagation

Unlike query-time spreading activation (used by recall), deja_vu needs a lightweight activation pass that runs whenever new context enters the agent. Incoming context activates seed nodes, activation spreads outward along weighted edges (bounded to 2 hops), and if any node's activation exceeds the recognition threshold, a match fires.

Performance note: Two hops on a pruned graph with a few thousand nodes is milliseconds, not seconds. The propagation is bounded BFS with weight thresholds, not a full graph traversal.

2. Dynamic Edge Weights

In a standard knowledge graph, the edge between "TechCo" and "healthcare" has a fixed weight. In the adaptive graph, edge weights change based on:

  • Usage frequency — edges traversed during successful retrievals get stronger
  • Temporal decay — unused edges weaken over time (exponential + power-law hybrid)
  • Outcome reinforcement — edges associated with correct predictions get a dopamine boost
  • Negative feedback — edges associated with false positives (unnecessary deja_vu triggers) get dampened

3. Subgraph Pattern Recognition

Simple deja_vu is "I've seen this entity before." Interesting deja_vu is "I've seen this pattern before" — a small healthcare company applying for an Innovate UK grant with strong technical credentials but weak commercialisation narrative. That's a subgraph pattern, not a single node.

Two approaches (v1 → v2):

V1 — Structural Fingerprints

Ship now

Hash the key structural pattern (sector + funder + company_size + outcome) into a fingerprint. Store it. On new context, compute fingerprint, check for collisions. O(1) lookup, exact structural match only. Good enough for launch.

V2 — Learned Pattern Embeddings

Ship later

Embed the local subgraph around each evaluation into a dense vector — not just text, but structural relationships. New context gets its structural embedding and similarity search finds fuzzy subgraph matches. "This feels like that time..."

DejaVu Engine (Core Algorithm)

class DejaVuEngine:
    def __init__(self, graph, threshold=0.7):
        self.graph = graph
        self.base_threshold = threshold

    async def check(self, context, agent_state=None):
        # 1. Extract entities from incoming context
        seeds = self.graph.extract_entities(context)

        # 2. Activate seed nodes (base activation + residual from prior access)
        activations = {}
        for entity in seeds:
            node = self.graph.get_node(entity)
            if node:
                activations[node.id] = node.base_activation + node.residual

        # 3. Spread activation — bounded BFS, 2 hops max
        spread = self.graph.propagate(activations, max_hops=2)

        # 4. Modulate threshold (norepinephrine lowers it → more sensitive)
        threshold = self.base_threshold
        if agent_state:
            threshold *= agent_state.norepinephrine_factor

        # 5. Check for recognition (exclude seed nodes — don't match input itself)
        matches = [
            (node_id, activation)
            for node_id, activation in spread.items()
            if activation > threshold and node_id not in seeds
        ]

        if not matches:
            return None

        # 6. Retrieve memory associated with strongest match
        best = max(matches, key=lambda x: x[1])
        memory = self.graph.get_memory(best[0])

        return DejaVuMatch(
            memory=memory,
            activation=best[1],
            pattern_type=self.classify_match(seeds, best[0]),
        )

06 — Voice Architecture

Layered Response Model

The voice interface uses temporal model routing — the fast model isn't a fallback, it's a bridge. It holds the conversation while the real thinking happens behind it.

The Layering Problem (Solved)

Layer 1 — STT

Deepgram

Speech-to-text. ~200-300ms latency. Cheap at scale. Streams partial transcripts for early intent classification.

Layer 2 — Bridge Response

Gemini Flash

Instant acknowledgment. <500ms. Generates a contextual bridge response ("Let me check that..." / "Good question about...") that starts speaking immediately.

Layer 3 — Reasoning

Claude Sonnet/Opus

Full reasoning. 2-3s. Runs in parallel with the bridge. When ready, the voice output crossfades from the bridge to the real answer seamlessly.

Layer 4 — TTS

ElevenLabs

Text-to-speech with native interrupt handling. When the reasoning model's answer arrives, ElevenLabs handles the smooth crossfade from the bridge response.

Language Primitive: stream...then...

Drift expresses temporal model routing as a language-level construct. The transpiler generates concurrent execution — the bridge and reasoning models run in parallel.

agent Jarvis {
  model: stream "gemini-flash" then "claude-sonnet"
  voice: eleven_labs(voice: "jarvis", interrupts: true)
  memory: dendric("jarvis_core")

  step respond(utterance: string) {
    -- Flash fires immediately, starts speaking
    let quick = generate acknowledgment as string
    respond quick

    -- Meanwhile, the real reasoning happens
    recall context from memory using utterance

    deja_vu match on utterance {
      "recurring_question" -> {
        respond "You asked about this last week — {match.previous_answer}"
        return
      }
    }

    let answer = answer utterance using context as string
    respond answer

    remember answer tagged utterance.intent
  }
}

Cost Model

50-100 voice interactions per day at heavy use. Each: Deepgram STT (fractions of a cent), one Flash call (fractions of a cent), one Sonnet call (a few cents), one ElevenLabs TTS (pro tier). 07 — Personal Agent Architecture

The Jarvis Pattern

The full system — Drift language + Dendric memory + voice interface — enables a personal AI agent that thinks, remembers, and speaks. The architecture is a multi-agent graph where every node is a .drift file sharing a unified Dendric memory.

System Architecture

Voice Input (Deepgram)
↓
Intent Router (Drift agent — Haiku)
↓ classifies intent →
Calendar Agent
Pintail Agent
Grant Agent
Compose Agent
↕ all share ↕
Dendric Memory (shared across all agents)
↓
Voice Output (ElevenLabs)

Key Properties

  • Every box is a .drift file. The intent router is itself a Drift agent — classify utterance as Intent — running on Haiku because intent classification is cheap.
  • Shared memory creates unified intelligence. When the grant checker learns about a company, the compose agent can reference it when drafting an email. When Pintail spots a rare aircraft, the reasoning agent can connect it to the defense news you read that morning.
  • deja_vu in voice is the uncanny moment. The agent interrupts — not because you asked, but because spreading activation fired. "Hey, this funder rejected a similar application last month. The feedback was about weak commercialisation." You didn't ask. The agent noticed.
  • Feedback trains the system. "Good catch" → dopamine signal → strengthen the edges that triggered recognition. "Yeah I know" → nudge the threshold up for that pattern type. The system learns what's worth flagging without explicit configuration.

Hardware Roadmap

Now — API-based

Everything runs on APIs

Deepgram, Gemini Flash, Claude, ElevenLabs. Works today. £3-5/day at heavy use. Drift's budget primitive manages cost automatically.

Later — Local inference

Blackwell GPU(s) + custom PC

Move STT and flash model local. Shave network latency and eliminate API costs for the bridge layer. Frontier reasoning stays API-based (or local Opus-class when hardware allows). Always-on agent running through Drift.

Anthropic's Model Context Protocol (MCP) is becoming the standard for how agents interact with external tools. If Drift has native MCP support, every MCP server in the ecosystem becomes a Drift tool automatically.

agent Notifier {

09 — Multi-Agent Composition

Agents Calling Agents

Real systems aren't one agent — they're networks. Drift makes inter-agent calls look like step calls, with the parallel keyword for concurrent fan-out.

agent GrantPipeline {
  memory: dendric("pipeline_history")

  step process(company: document, calls: list<document>) -> Report {
    -- Fan out: check each grant call in parallel
    let scores = []
    for each call in calls parallel {
      let score = GrantChecker.evaluate(company, call)
      scores.add(score)
    }

    -- Synthesize results
    let report = summarize scores as Report

    remember report tagged company.name, "pipeline_run"
    return report
  }
}

One agent calling another agent's steps. The parallel keyword means they fan out concurrently. The runtime handles orchestration, error propagation, and cost aggregation across agents. A pipeline that would be 200+ lines of LangGraph reads like English.

10 — Distribution Strategy

Getting This Into the World

Why Languages Fail (and How This One Won't)

Almost all new programming languages die. The ones that survive have one of three things: a killer app that forces adoption (Ruby → Rails), institutional backing (Go → Google), or they start as a narrow DSL in a specific domain and expand outward (SQL, Terraform, HCL). Drift targets the third path — start as the agent DSL, expand from there.

14 — The Thesis, Restated

What This Is

Drift + Dendric is not a framework, not a library, not a tool. It's a cognitive architecture for agents, expressed as a programming language.

Drift is how an agent thinks — intent verbs, cost-aware model routing, structured output, progressive disclosure from English-adjacent to full engineering control.

Dendric is how an agent remembers — four retrieval paths, temporal decay, spreading activation, and the deja_vu primitive that lets agents recognise patterns they weren't looking for.

The voice architecture is how an agent speaks — layered temporal model routing that bridges the gap between instant acknowledgment and thoughtful reasoning.

No other system has all three designed together from first principles. That co-design is the moat. The memory system knows about cost. The language knows about memory. The voice interface knows about both. They share assumptions, they share constraints, they share a philosophy: agents are non-deterministic, they have cost, they fail gracefully, they remember, they learn, they compose.

The transpiler works today. Three .drift files transpile to Python and execute end-to-end. The Dendric spec is complete with benchmarks. The voice stack uses production APIs. This isn't a whiteboard. This is buildable, and the spec you're reading is the build plan.
